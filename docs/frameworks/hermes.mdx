# Hermes plugin

The Hermes plugin (`plugins/stacks/hermes`) exposes the same **19 core Stacks tools** to Python-based Hermes agents. Every tool call — read and write — delegates to `@sugarhi11/agent-core` through a Node.js subprocess bridge.

## Installation

<Steps>
  <Step title="Install Node bridge dependencies">
    ```bash
    cd plugins/stacks/hermes
    npm install
    ```
    This installs `@sugarhi11/agent-core` used by `scripts/stacks-bridge.mjs`.
  </Step>
  <Step title="Link the plugin">
    ```bash
    ln -s "$(pwd)/plugins/stacks/hermes" ~/.hermes/plugins/stacks
    ```
    Or copy the `hermes/` directory into your Hermes plugins folder.
  </Step>
  <Step title="Configure environment">
    Set variables in `~/.hermes/.env`:

    ```bash
    STACKS_NETWORK=testnet
    STACKS_SENDER_KEY=your_testnet_private_key_hex
    ```
  </Step>
  <Step title="Enable in Hermes config">
    Add `stacks` to your enabled plugins list in your Hermes configuration file.
  </Step>
</Steps>

No pip packages are required — see `requirements.txt`.

## Environment variables

| Variable | Purpose |
| --- | --- |
| `STACKS_NETWORK` | `mainnet` or `testnet` — injected into tool params when omitted |
| `STACKS_SENDER_KEY` | Hex private key for write tools — injected when `senderKey` is absent |
| `STACKS_WALLET_ADDRESS` | Optional explicit wallet address |

Read-only tools work without `STACKS_SENDER_KEY`. Write tools register a `check_fn` that verifies signing is configured before the handler runs.

## Architecture

```
┌─────────────────┐
│  Hermes agent   │
│  (Python)       │
│                 │
│  tools.py       │     subprocess
│       │         │ ──────────────► stacks-bridge.mjs
│  bridge.py      │                      │
└─────────────────┘                      ▼
                                  @sugarhi11/agent-core
                                        │
                                        ▼
                                   Hiro / ALEX / Allbridge / chain
```

### Bridge dispatch

`bridge.py` spawns Node with `scripts/stacks-bridge.mjs`:

```bash
node stacks-bridge.mjs stacks_get_balance '{"address":"ST1..."}'
```

The bridge:

1. Loads `@sugarhi11/agent-core` and looks up the handler in `STACKS_TOOLS`
2. Injects `STACKS_NETWORK` and `STACKS_SENDER_KEY` from the environment when missing from params
3. Returns `{ success: true, result: ... }` or `{ success: false, error: "..." }`

A legacy write-only script (`stacks-write.mjs`) remains for backward compatibility; new integrations should use `stacks-bridge.mjs`.

## Plugin features

Beyond the 19 tools, the Hermes plugin registers:

| Feature | Description |
| --- | --- |
| **`/stacks`** | Slash command showing wallet/network status and signing readiness |
| **`pre_llm_call` hook** | Injects wallet context on the first turn of each session |
| **`post_tool_call` hook** | Logs Stacks tool usage |
| **Bundled skill** | `skills/stacks-onchain/SKILL.md` — load via `skill_view("stacks:stacks-onchain")` |

## Write tools

These tools require Node.js 22+ on `PATH` and `STACKS_SENDER_KEY`:

| Tool | Description |
| --- | --- |
| `stacks_send_tokens` | STX transfer |
| `stacks_stack` | Lock STX for PoX |
| `stacks_delegate_stx` | Delegate to a pool |
| `stacks_revoke_delegate` | Revoke delegation |
| `stacks_transfer_name` | Transfer BNS name |
| `stacks_contract_call` | Clarity contract call |
| `stacks_swap_execute` | Execute ALEX swap |
| `stacks_bridge_initiate` | Start bridge transfer |

## Tool schemas

JSON schemas in `schemas.py` mirror the agent-core contract so the LLM sees identical tool definitions across ElizaOS, OpenClaw, and Hermes.

## Error handling

All handlers return a consistent JSON envelope:

```json
{ "success": false, "error": "Failed to fetch balance: 404 Not Found" }
```

Bridge timeouts default to **120 seconds**.

<Warning>
  The bridge runs with `cwd` set to `plugins/stacks/hermes/`. Deploy Hermes with the plugin directory intact (including `node_modules/@sugarhi11/agent-core` and `scripts/stacks-bridge.mjs`).
</Warning>

## Amount parsing

Hermes adapters accept human-readable STX amounts (e.g. `"1 STX"`, `"0.5"`) and convert them to microSTX before calling agent-core handlers.
